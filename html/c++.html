선언(declaration)과 정의(definition)

🔗MSDN C++ Reference: Declarations and Definitions

· 기초 공사 부터 튼튼하게! 선언(declaration)과 정의 (definition)부터 알아보자!

· 토큰(token): 한 코드에서 빈칸, 캐리지리턴으로 더 이상 분리할 수 없는 요소(괄호 자체, 변수명, 함수명, 자료형 등...)
· 식별자(identifier): 사용자가 지정한 이름 (변수명, 함수명, 클래스 명 등...)

선언(declaration)

· '메모리의 할당 여부'로 파악할 수 있다.
· 선언(declaration): 컴파일러에게 이 코드를 사용하기 위해 참조할 식별자(identifier)와 이름을 알리는 것 (메모리를 할당하지 않음)

정의(definition)

· 정의(definition): 컴파일러에게 이 코드를 위한 메모리를 할당하라고 알리는 것 (메모리가 할당 됨) 따라서 정의는 반드시 하나만 존재해야 한다. (odr: 유일 정의 원칙)

· 예
· 함수는 몸체의 유무로 선언(원형)과 정의가 나뉜다.
· 변수는 선언과 동시에 정의이다. 선언함과 동시에 정의가 되는 것이다. 선언함과 동시에 정의가 되기 때문에 초반에 선언, 정의를 혼동하기 쉽다.

변수의 선언

· 변수에 있어서 선언과 정의는 처음에 조금 햇갈린다. 여기서 확실이 집고 넘어가자

· 변수의 선언: 정의 선언, 정의라고도 하며, 통상적으로 변수의 선언, 변수의 정의 둘 중 하나를 쓴다.
· 일반적으로 쓰는 단어와 달리 사실 변수는 선언과 동시에 정의된다.
· 변수는 사용되기 위해 컴파일러에게 알려져야 해서 선언해야하고 선언하면 메모리가 할당되어 사용가능 하므로 변수로서 의미(정의)를 가지게된다.
· 메모리가 할당 되기 때문에 유일하게 정의되어야한다. 만약 유일 정의 원칙이 없다면, 컴파일러가 같은 이름을 보고 무슨 메모리를 써야할지 모른다.


· 변수의 참조 선언: 선언을 했지만 정의가 안되는(엄밀히 말하면 사실 이미되어있는) 경우도 있다.

변수의 선언과 동시에 초기화

· C++은 변수를 선언하자마자 초기화(전역이나, 정적 변수는 자동으로 0으로 초기화되긴 함)해야한다. 값을 지정(즉 선언과 동시에 초기화)하지 않으면
· 변수를 선언하여 변수에 할당된 메모리 블록에는 그전에 쓰다가 버린 비트들(=쓰레기 값: garbage value)이 그대로 있다.
· 이것을 자료형에 의거에 해석하면 의도치 않은 이상한 값이 나올 것이다.

· 초기화를 하지 않으면 대부분의 컴파일러가 경고를 출력한다.
· 변수의 초기화 방법에는 3가지가 있다.

    register 변수 cpu register, region, auto, no-linckage

    · CPU의 레지스터, 지역 스코프, 자동 생명주기, 링크 없음
    ㆍregister 키워드는 C에서는 자동 변수를 CPU의 register에 저장하는 기능이었다.
    ㆍregister 키워드는 C++11이전 버전에서는 H/W와 컴파일러가 매우 정밀하게 개발된 경우를 제외하고는
    ㆍ"자주 사용되는 변수" 혹은 "특별한 조치가 필요한 변수"의 의미로 사용되었다.
    ㆍregister 키워드는 C++11이후 버전에서는 자동 변수임을 명시하는데 사용하며,
    ㆍ오로지 자동 변수만 사용하는 경우에는 사용자가 전역 변수와 동일한 이름을 지닌 자동 변수(지역 변수)를 구분짓는 의미로 쓰이기도 한다.
    ㆍregister키워드의 의미를 대체 하기 위해 auto키워드를 쓸 수 있지만,
    ㆍ기존 코드가 auto 키워드를 인식하지 못하는 경우를 대비하여 C++11이후 버전에서도 register 키워드가 계승되었다.

    전역 변수 data, global, static, outter-linkage
    extern 전역 변수
    (전역 변수의 명시적 표현)    		 same as global variable

    · 초기화 실시여부에 따라 2가지로 나뉨
    · 초기화 O: 정의, 위의 전역 변수와 같음
    · 초기화 X: 참조 선언, 외부 파일에 선언된 전역 변수를 가져와 쓰겠다라는 의미(단일 정의 원칙: '한 변수는 한 번만 선언' 준수위함임)
    · 즉, 여러 파일에 같은 이름의 전역 변수가 여러개 있다면, 반드시 단 하나는 선언이어야하고, 나머지들은 참조 선언이어야 한다.
    · 다르게 말해, 여러 파일에서 전역 변수가 사용된다면, 오직 하나의 파일만이 그 전역 변수에 대한 정의를 가진다. (유일 정의 원칙)
    · 여러 파일에 같은 이름의 전역 변수들이 모두 선언이면 odr(단일 정의 원칙)위배이고, 반대로 모두 참조 선언이면 참조할 변수가 없는 오류가 발생한다.
    · 참조 선언은 말 그대로 어떤 전역 변수를 참조하는 변수이므로 메모리 블록이 할당되지 않는다.

    static 전역 변수 data, global, static, inner-linkage

    · 선언시 0으로 자동 초기화
    · 데이터 영역, 전역 스코프, 정적 생명 주기, 내부 링크
    · static은 '정적'임을 나타내는데, 전역 변수는 이미 정적이다.
    · static이 전역 변수에 붙으면 이미 정적인 생명 주기 대신 링크를 외부 링크에서 내부 링크로 바꾼다고 이해하자.
    · 외부링크에서 내부링크로 변경되면 프로그램은 외부 선언으로 인식하려 하지 않기 때문에, 외부에 같은 이름의 변수가 선언되어있어도 상관없다.

    const 전역 변수 data, global, static, inner-linkage

    · 데이터 영역, 전역 스코프, 정적 생명 주기, 내부 링크
    · 즉, 전역 변수가 상수화 되면, 외부 링크가 내부 링크로 바뀜.
    · 내부 링크로 바뀌므로 외부에 같은 이름의 전역 변수가 선언되어도 상관 없음.
    · 따라서 #include를 사용하여 모든 const 전역 변수를 선언한뒤 프로그램의 각 파일들에 #include 해줄수 있음 (실제로 이렇게 많이 씀)
    · const 전역 변수는 상수이므로 선언시 초기화를 해줘야함.
    · 상수인 이유도 있지만, 내부 링크를 가지기 때문에 참조 선언이 없으므로 그냥 초기화 하면 되는 것임.
    · 어떤 이유로 const 전역 변수는 외부 링크를 가지게 하려면 무조건 선언에 extern을 붙여야함
    · 그리고 외부 파일에서도 extern을 이용하여 선언해야한다.

    · 변수를 선언하는 순간, 변수는 저장공간 위치, 생명주기, 스코프(사용범위), 링크를 가지게된다.
    · 지역 변수는 자동 생명주기를 가지기 때문에 스택 영역에 저장된다 생성과 소멸이 함수나 스코프에 들어갈 때 일어나기 때문이다.
    · 첨언하자면, 지역 변수의 저장과 같은 원리로 함수의 주소 또한 실행과 종료에 따라야하므로 스택 프레임이라는 곳에 저장된다.
    · 전역 변수는 정적 생명 주기를 가지므로 스택과 같은 구조가 필요 없이 그냥 데이터 영역에 저장만 하면 된다.
    · new, malloc()을 사용한 동적 메모리는 사용범위나 링크 규칙에 의해서가 아니라, new와 delete연산자에 의해 결정된다. (heap 영역)

    이름 설명
    auto

    · C++ 이전에는 어떤 변수가 자동 변수라는 선언에 사용되었음
    · C++11 이후는 자동 변수의 선언에 사용

    register

    · CPU의 레지스터 메모리를 사용할때 사용함
    · C++에서는 단순히 자동 변수의 명시적 표현에 사용됨

    static

    · 전역 변수에 사용: 외부 링크를 내부 링크로 변경한다라는 의미
    · 지역 변수에 사용: 자동 생명 주기를 정적 생명 주기로 변경한다는 의미

    extern

    · 참조 선언(초기화 미실시 시): 외부 파일에 이미 정의된 변수를 사용한다는 의미
    · 전역 변수의 명시적 표현(초기화 실시 시): 전역 변수의 명시적 표현이다.

    thread_local

    · 쓰레드 생명 주기: 변수의 생명 주기가 변수가 포함된 쓰레드의 존속시간이다.

    const

    · 변수가 한 번 초기화 된 이후로는 그 값을 변경할 수 없음을 의미한다.
    · 열거형인 enum을 사용하여 많은 상수에 이름을 붙여 쓸 수 있다.

    volatile

    · 프로그램 코드가 변경하지 않더라도 변수의 값이 변경될 수 있다라는 의미이다.
    · 컴퓨터와 연결된 어떤 하드웨어(예를 들어 시간 정보가 들어있는 타이머기계)가 직접 그 메모리의 값을 변경할 수도 있다.
    · 컴파일러의 변수 최적화를 막는다. 짧은 코드에서 같은 변수가 두번 사용되면 컴파일러는 값을 두 번 참조하지 않고 레지스터에서 그 값을 캐시한다.
    · 하지만 volatile을 사용할 경우 이런 최적화를 수행하는것을 막고 직접 메모리에서 값을 참조한다.

    rvalue 참조형
    lvalue, rvalue 란?

    · C부터 내려온 것으로 일반적으로 대입 연산자(=)의 피연산자 중 좌항을 lvalue 우항을 rvalue라고 한다.
    · 틀린이야기는 아니나, 다른 관점에서 볼 필요가 있다.

    C++에서 lvalue, rvalue란

    · lvalue는 메모리상에서 지속적인 주소를 가진 값이다. 즉, 변수인데 표현식(expression) 이후에도 없어지지 않는다.
    · rvalue는 지속적인 메모리 주소와 관계 없는 값이다. 즉, 숫자나 표현식(2, x + 2 등...), 표현식 이후에는 사라지는 값이다.

    함수의 저장공간, 생명 주기, 링크

    · C++에서 함수는 다른 함수 안에 정의할 수 없다. 따라서 모든 함수는 정적 생명주기이다.
    · 함수는 디폴트가 외부 링크를 가진다. 따라서 여러 파일에서 함수를 공유할 수 있다.
    · 함수 원형에 extern 키워드를 붙이면 다른 파일에 정의되어있음을 나타낼 수 있으나 그것은 선택사항이다.
    · static 키워드를 함수 원형, 정의 둘 다 작성하면 내부 링크가 부여된다.
    · 내부 링크가 부여되면 같은 이름을 가지는 함수를 외부에서 정의할 수 있다.
    · 파일 내부에서는 외부 같은 이름 함수를 무시하고 파일 내부의 정적 함수를 사용한다.
    · 단일 정의 원칙에 의해 함수도 인라인이 아니면 프로그램 내에서 단 하나의 정의만 가져야한다.
    · 인라인 함수는 단일 정의 원칙을 따르지 않는다. 따라서 헤더 파일에 인라인 함수 정의를 넣어도된다.
    (#include 되면 각 파일에 그 코드가 복사되니까 결국 각 파일이 인라인 함수 정의를 가지게 됨)
    · 프로그램의 코드 파일들 중 이 헤더 파일을 포함한다면 각각은 모두 인라인 함수 정의를 가지게되는 것이다.
    · 그러나 C++은 특정 인라인 함수의 정의가 모두 동일해야함을 요구한다.

    언어 링크(language linking)

    람다(Lamda)

    · [캡처 변수] (매개변수) mutable -> 반환형 { 함수내용 };의 형태

    [캡처 변수]

    · 람다 밖의 변수를 말그대로 캡처해서 람다 안에서 씀, 캡처할 밖의 변수명과 동일하게 적어야함
    · 캡처할 변수가 없더라도 빈 []를 반드시 적어야됨 이것이 람다의 시작을 알림

    (매개변수)

    · 람다 안에서 사용하는 파라미터

    mutable

    · 생략 가능
    · [캡처 변수]안의 변수는 기본적으로 람다 외부 변수이므로 읽기 전용이 됨
    · 따라서 람다 안에서 캡처 변수에 값을 대입하지 못함
    · mutable을 사용하면 2가지 기능이 있음
    · 1. 람다 안에서 캡처 변수가 복제됨, 그래서 읽기 쓰기는 가능하지만, 밖의 변수에는 영향X, 즉 일반 함수의 값으로 전달 처럼 됨
    · 2. 클로저 변수: 캡처 변수가 복제되고 람다가 종료되도 복제 된 캡처 변수는 '람다 내부에서만은' 그 값을 유지함

    -> 반환형

    · 반환형으로 void, 특정데이터형 생략 가능
    · -> 반환 형식은 말 그대로 반환 형식, 생략 가능 void이면 void로 특정 데이터형이면 알아서 추론함

    클래스

    클래스

    · 복합 자료형으로, 사용자가 직접 정의하여 사용할 수 있다. 자료형 중 가장 복잡하다.
    · C++의 목표는, 클래스 사용을 마치 기본 내장 데이터형을 사용하는 것과 비슷하게 하는 것이다.

    객체 지향 5대 원칙(SOLID)

    접근 제한자

    이름 설명
    public
    • · 클래스 외부에서 읽기 쓰기 가능
    • · 클래스의 객체와 프로그램 사이의 인터페이스 역할
    • · 설계의 추상화가능 세부적인 구현을 추상화(private)와 분리하는 것을 캡슐화(encapsulation)라고 함
    • · 데이터 은닉은 캡술화의 한 예
    • · 즉, 클래스는 기본적으로 private에는 추상화한 것(사람이라면 눈, 코, 입)
    • · public에는 추상화 한것의 세부 기능(눈 깜빡임, 코로 냄새 맡기, 입으로 말하기)
    private
    • · 클래스의 디폴트 접근 제한자(참고로 구조체는 public이 기본 접근 제한)
    • · 클래스 내부에서만 읽기 쓰기 가능
    • · 외부에서 임의로 접근 불가능하게 함, 데이터 은닉(data hiding)
    • · 데이터의 무결성을 지킬 수 있으므로 좋은 것
    protected
    • · 상속 받은 클래스에서 읽기 쓰기 가능

    생성자와 소멸자

    이름 설명
    디폴트 생성자
    • · 디폴트 생성자 3가지
    • · 아예 정의하지 않으면 자동 생성
    • · 매개변수가 없는 생성자를 직접 오버로딩
    • · 디폴트 매개변수만 있는 생성자를 직접 오버로딩
    • · 디폴트 생성자는 클래스에 단 하나만 존재해야함, 만약 저 방법을 다 써서 만들면 안됨
    사용자 정의 생성자
    • · 사용자가 직접 정의한 생성자
    • · 생성자는 반환형이 아예 없음(void도 X) 클래스 이름과 동일한 이름으로 작성
    • · 일반적으로 멤버 변수는 private에 선언하므로 객체 생성 후 객체를 통해 초기화 불가
    • · 객체를 성공적으로 초기화 하기위한 적당한 멤버 함수가 바로 생상저임
    • · 생성자 내부에 멤버 변수 초기화, 파라미터를 정의하여 직접 값을 전달, 디폴트 파라미터를 정의하거나 하면 됨
    • · 생성자를 사용하는 2가지 방법
    • · 명시적 호출
      • (코드 추가) A a = A(); A* a = new A();
    • · 암시적 호출
      • (코드 추가) A a; // A a(); 이거는 틀림! 주의 이거는 함수 선언표현이되어버림
      A* a = new A; A* a = new A;
    • · 객체의 동적 메모리 대입
      • (코드 추가)
    디폴트 복사 생성자
    • 정의하지 않으면 자동 생성

    멤버

    이름 설명
    멤버 변수
    • · 일반적으로 private에 선언
    • · 데이터 멤버라고도 함
    멤버 함수
    • · 일반적으로 public에 선언
    • · 멤버 함수는 정의할 때, 사용 범위 결정 연산자가 있어야 함, 그 함수가 어느 클래스에 속하는지 나타내야한다.
    • · 멤버 함수 안에서는 클래스의 private 멤버에 접근 가능
    • · 멤버가 아닌 함수안에서 클래스 private 멤버에는 접근 불가, 하지만 friend 함수는 가능
    • · 클래스 선언 안에 멤버 함수 정의가 있다면 자동으로 인라인 함수가 됨
    • · 코드 수정 규칙(rewrite rule)에 따르면 이는 선언 안의 정의는 원형으로 대체되고, 클래스 선언 바로 뒤에 정의가 추가되는 것과 같음
    • · 클래스 선언 밖에 정의한 경우도 멤버 함수 선언 앞에 inline 키워드를 붙이면 인라인 함수가 됨
