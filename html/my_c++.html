변수

· 문법: (제한자) (데이터형) (변수명)
· RAM(Random Access Memory)의 메모리 블록에 값을 할당하여 사용하기 위함.
· (제한자): 변수에 특정 기능을 부여
· (데이터형): 변수명으로 할당된 메모리 블록의 비트값을 어떤 값으로 해석할지 결정
· (변수명): 변수로 지정된 메모리 블록에 붙여주는 이름

선언(Declaration)과 정의(Definition)

토큰(Token)

· 한 코드에서 빈칸, 캐리지 리턴(Enter키의 입력)으로 더 이상 분리할 수 없는 요소 (괄호 그 자체, 변수명, 함수명, 자료형등; 예를 들어 자료형 int는 in t 혹은 i nt로 쓸 수 없다. )

식별자(identifier)

· 프로그래머가 직접 입력하여 지정한 이름들 (변수명, 함수명, 클래스명 ...)

선언(Declaration)

· 컴파일러에게 이 코드를 사용하기위해 참조할 식별자(Identifier)와 이름을 알리는 것(메모리를 할당하지 않음)

정의(Definition)

· 컴파일러에게 이 코드를 위한 메모리를 할당하라고 알리는것 (즉, 메모리가 할당된),
· 따라서 정의는 반드시 하나만 존재해야함 (ODR: 유일 정의 원칙)

· 변수를 본다면, 변수는 선언과 동시에 정의이다. 선언함과 동시에 정의가 되는 것이다. 선언함과 동시에 정의가 되기 때문에 초반에 선언, 정의를 혼동하기 쉽다.

변수의 선언과 정의

변수의 선언

· 정의 선언, 정의 라고도 한다.
· 통상적으로 변수의 선언, 변수의 정의 둘 중 하나를 사용한다.
· 통장석으로 사용하는 단어와 달리 사실 변수는 선언과 동시에 정의된다.
· 변수를 사용하기 위해 선언하면, 그와 동시에 이후에는 사용이 가능해야하므로 메모리가 할당된다.
· 즉, 변수로서의 의미를 가지므로 (정의) 변수는 선언과 동시에 정의된다.

변수의 참조 선언

· 변수가 선언되었지만 정의는 안되는(엄밀히 말하면 사실 이미 되어있는)것.
· extern 전역 변수가 그 예이다.

· extern int a;
· 외부 링크를 가지는 현재 파일 밖의 다른 파일에 있는 정수형 변수 a를 현재 파일에서 사용하고자할 때
· 이는 외부 파일에서 이미 정의된 경우이므로 선언만 된다.

변수의 선언과 동시에 초기화

· 변수는 선언과 동시에 초기화(전역 정적 변수는 자동으로 초기화 되긴 한다. )해야한다.
· 값을 지정(즉, 선언과 동시에 초기화)하지 않으면, 쓰레기값(Garbage Value; 전에 쓰다 버린 비트 값)이 그대로 있게된다.
· 쓰레기 값을 자료형에 의거하여 해석하면 의도치 않은 이상한 값이 나올 것이다.

· 초기화를 하지 않으면 대부분의 컴파일러가 경고를 표시한다.
· 변수 초기화 방법 3가지는 아래와 같다.

    extern 전역 변수
    (전역 변수의 명시적 표현)    		 same as global variable

    · 초기화 실시여부에 따라 2가지로 나뉨
    · 초기화 O: 정의, 위의 전역 변수와 같음
    · 초기화 X: 참조 선언, 외부 파일에 선언된 전역 변수를 가져와 쓰겠다라는 의미(단일 정의 원칙: '한 변수는 한 번만 선언' 준수위함임)
    · 즉, 여러 파일에 같은 이름의 전역 변수가 여러개 있다면, 반드시 단 하나는 선언이어야하고, 나머지들은 참조 선언이어야 한다.
    · 다르게 말해, 여러 파일에서 전역 변수가 사용된다면, 오직 하나의 파일만이 그 전역 변수에 대한 정의를 가진다. (유일 정의 원칙)
    · 여러 파일에 같은 이름의 전역 변수들이 모두 선언이면 odr(단일 정의 원칙)위배이고, 반대로 모두 참조 선언이면 참조할 변수가 없는 오류가 발생한다.
    · 참조 선언은 말 그대로 어떤 전역 변수를 참조하는 변수이므로 메모리 블록이 할당되지 않는다.

    const 전역 변수
    • data
    • global
    • static
    • inner-linkage

    · 데이터 영역, 전역 스코프, 정적 생명 주기, 내부 링크
    · 즉, 전역 변수가 상수화 되면, 외부 링크가 내부 링크로 바뀜.
    · 내부 링크로 바뀌므로 외부에 같은 이름의 전역 변수가 선언되어도 상관 없음.
    · 따라서 #include를 사용하여 모든 const 전역 변수를 선언한뒤 프로그램의 각 파일들에 #include 해줄수 있음 (실제로 이렇게 많이 씀)
    · const 전역 변수는 상수이므로 선언시 초기화를 해줘야함.
    · 상수인 이유도 있지만, 내부 링크를 가지기 때문에 참조 선언이 없으므로 그냥 초기화 하면 되는 것임.
    · 어떤 이유로 const 전역 변수는 외부 링크를 가지게 하려면 무조건 선언에 extern을 붙여야함
    · 그리고 외부 파일에서도 extern을 이용하여 선언해야한다.

    · 변수는 선언하는 순간, 저장곤간, 생명주기, 스코프(사용범위), 링크를 가지게 된다.
    · 역 변수는 자동 생명주기를 가지기 때문에 스택 영역에 저장된다 생성과 소멸이 함수나 스코프에 들어갈 때 일어나기 때문이다.
    · 첨언하자면, 지역 변수의 저장과 같은 원리로 함수의 주소 또한 실행과 종료에 따라야하므로 스택 프레임이라는 곳에 저장된다.
    · 전역 변수는 정적 생명 주기를 가지므로 스택과 같은 구조가 필요 없이 그냥 데이터 영역에 저장만 하면 된다.
    · new, malloc()을 사용한 동적 메모리는 사용범위나 링크 규칙에 의해서가 아니라, new와 delete연산자에 의해 결정된다. (heap 영역)

    제한자(storage class specifier)

    이름 설명
    auto

    · C++ 이전에는 어떤 변수가 자동 변수라는 선언에 사용되었음
    · C++11 이후는 자동 변수의 선언에 사용

    register

    · CPU의 레지스터 메모리를 사용할때 사용함
    · C++에서는 단순히 자동 변수의 명시적 표현에 사용됨

    static

    · 전역 변수에 사용: 외부 링크를 내부 링크로 변경한다라는 의미
    · 지역 변수에 사용: 자동 생명 주기를 정적 생명 주기로 변경한다는 의미

    extern

    · 참조 선언(초기화 미실시 시): 외부 파일에 이미 정의된 변수를 사용한다는 의미
    · 전역 변수의 명시적 표현(초기화 실시 시): 전역 변수의 명시적 표현이다.

    thread_local

    · 쓰레드 생명 주기: 변수의 생명 주기가 변수가 포함된 쓰레드의 존속시간이다.

    mutable

    · 특정 구조체가 const로 선언되어있어도, 구조체의 특정 멤버를 변경할수 있음을 나타낸다.

    const

    · 변수가 한 번 초기화 된 이후로는 그 값을 변경할 수 없음을 의미한다.
    · 열거형인 enum을 사용하여 많은 상수에 이름을 붙여 쓸 수 있다.

    volatile

    · 프로그램 코드가 변경하지 않더라도 변수의 값이 변경될 수 있다라는 의미이다.
    · 컴퓨터와 연결된 어떤 하드웨어(예를 들어 시간 정보가 들어있는 타이머기계)가 직접 그 메모리의 값을 변경할 수도 있다.
    · 컴파일러의 변수 최적화를 막는다. 짧은 코드에서 같은 변수가 두번 사용되면 컴파일러는 값을 두 번 참조하지 않고 레지스터에서 그 값을 캐시한다.
    · 하지만 volatile을 사용할 경우 이런 최적화를 수행하는것을 막고 직접 메모리에서 값을 참조한다.

    lvalue, rvalue

    · C부터 내려온 것으로 일반적으로 대입 연산자(=)의 피연산자 중 좌항을 lvalue 우항을 rvalue라고 한다.
    · 틀린이야기는 아니나, 다른 관점에서 볼 필요가 있다.

    C++에서 lvalue, rvalue란?

    · lvalue는 메모리상에서 지속적인 주소를 가진 값이다. 즉, 변수인데 표현식(expression) 이후에도 없어지지 않는다.
    · rvalue는 지속적인 메모리 주소와 관계 없는 값이다. 즉, 숫자나 표현식(2, x + 2 등...), 표현식 이후에는 사라지는 값이다.

    rvalue 참조형

    · 말 그대로 rvalue를 참조하는 데이터형이다.
    · 엠퍼샌드(&) 연산자 2개를 이어쓴다.
    · (코드 추가) int&& a = 3; // rvalue인 3을 참조 하지만 이렇게 쓰지 않는다.
    · 클래스의 이동 생성자, 이동 대입 연산자에서 쓰인다. 뒤에 가서 더 자세히 다뤄보자.

    함수 원형의 반환형

    · 컴파일러에게 리턴값이 저장된 메모리를 몇 바이트를 읽고, 어떻게 해석할지 알려준다.
    · 프로그래머에게는 '리턴값을 바르게 처리하니까 걱정하지마!' 를 보장해준다.

    함수 원형의 매개변수 자료형

    · 반환형과 같이 컴파일러에게 어떤 매개 변수를 가지는지 알려준다.
    · 프로그래머에게는 '컴파일러가 정확한 갯수, 데이터형을 썼고 데이터형이 다르면 변환 가능하면 변환할게(자동 변환)' 를 보장해준다.

    함수의 저장 공간, 생명 주기, 링크

    · C++에서 함수는 다른 함수 안에 정의할 수 없다. 따라서 모든 함수는 정적 생명주기이다.
    · 함수는 디폴트가 외부 링크를 가진다. 따라서 여러 파일에서 함수를 공유할 수 있다.
    · 함수 원형에 extern 키워드를 붙이면 다른 파일에 정의되어있음을 나타낼 수 있으나 그것은 선택사항이다.
    · static 키워드를 함수 원형, 정의 둘 다 작성하면 내부 링크가 부여된다.
    · 내부 링크가 부여되면 같은 이름을 가지는 함수를 외부에서 정의할 수 있다.
    · 파일 내부에서는 외부 같은 이름 함수를 무시하고 파일 내부의 정적 함수를 사용한다.
    · 단일 정의 원칙에 의해 함수도 인라인이 아니면 프로그램 내에서 단 하나의 정의만 가져야한다.
    · 인라인 함수는 단일 정의 원칙을 따르지 않는다. 따라서 헤더 파일에 인라인 함수 정의를 넣어도된다.
    (#include 되면 각 파일에 그 코드가 복사되니까 결국 각 파일이 인라인 함수 정의를 가지게 됨)
    · 프로그램의 코드 파일들 중 이 헤더 파일을 포함한다면 각각은 모두 인라인 함수 정의를 가지게되는 것이다.
    · 그러나 C++은 특정 인라인 함수의 정의가 모두 동일해야함을 요구한다.

    전달 방법 설명
    pass by value

    · 실인수(인수)는 값이고, 그 값이 가인수(매개변수)에 할당되어 함수에 전달

    pass by address

    · 실인수(인수)는 주소이고, 그 주소값이 가인수(매개변수)에 할당되어 함수에 전달
    · 즉, 값으로 전달인데 그 값이 주소임
    · 메모리의 주소가 가인수에 할당되므로 실인수와 가인수가 서로 연결

    pass by reference

    · pass by const reference: 참조로 전달하지만 함수안에서 변경을 하기 싫다면 이걸 사용

    [캡처 변수]

    · 람다 밖의 변수를 말 그대로 캡처하여 람다 안에서 사용
    · 캡처할 밖의 변수명과 동일해야함
    · 캡처할 밖의 변수가 없으면 []로 적어야함 []가 람다의 시작을 알림

    (매개 변수)

    · 람다 안에서 사용하는 파라미터

    mutable

    · 생략 가능
    · [캡처 변수]안의 변수는 람다 외부 변수이므로 읽기 전용이 됨
    · 따라서 람다 안에서 캡처 변수에 값을 대입하지 못함
    · mutable은 2가지 기능을 제공함

    · 1. 람다안에서 캡처 변수가 복제됨, 그래서 읽기 쓰기는 가능하나, 밖의 변수에는 영향이 없음, 즉 일반 함수의 값으로 전달과 같음
    · 2. 클로저 변수: 캡처 변수가 복제되고 람다가 종료되도 복제된 캡처변수는 '람다 내부에서만은' 그 값이 유지됨

    -> 반환형

    · 생략 가능
    · void, 반환 데이터형 둘 중 하나를 적으며, 반환 데이터형이 있어도 생략가능하며, 알아서 추론함

    #include <iostream>
     
    using namespace std;
     
    int main()
    {
     
        int num1 = 2;
        int num2 = 3;
     
        auto sum_l1 = [](int a, int b)->int { return a + b; };
        //auto sum_l2 = [num1](int a) { num1 += a; }	// 오류 num1은 읽기 전용이므로 값 할당 불가
        auto m_sum_l2 = [num1](int a) mutable { num1 += a; cout << num1 << endl; };		// 가능 mutable 키워드가 num1을 복제하여 람다에 전달, 복제된 num1는 값 할당 가능, 람다가 종료되어도 '람다 내부에서만' 그 값이 유지
        auto sum_l3 = [&num2](int a) { num2 += a; };	// 캡처 참조 변수, 참조형 이므로 람다 내부에서 캡처 변수로 외부 변수의 값 변경 가능

        m_sum_l2(2);	// 5

    🔗(예전 블로그) 클래스 정리 부분

    · 복합 자료형으로, 사용자가 직접 정의하여 사용할 수 있다. 자료형 중 가장 복잡하다.
    · C++의 목표는, 클래스 사용을 마치 기본 내장 데이터형을 사용하는 것과 비슷하게 하는 것이다.
    · 기본적으로 클래스는 선언이다 즉, 컴파일러에게 이런 클래스가 있다고 알려질 뿐, 객체화 되기전까지는 간단히 말해 메모리가 할당되지 않는다.

    객체 지향의 개념

    · 추상화: 현실 세계의 대상을 중요한 것만 추려 추상화 하여 클래스로 표현한다.
    · 데이터 은닉: 클래스 데이터를 private에 숨겨 클래스 외부에서 데이터에 직접 접근 할 수 없고, public 인터페이스를 통해서만 접근하도록 하는 것
    · 캡슐화: 클래스를 사용하여 멤버를 클래스 안에 묶는 것을 말한다. 따라서 클래스의 내부 데이터, 작동 방식등을 사용자가 알 수 없도록 하는 것이다.
    · 다형성: 함수의 재정의 (override), 동적 바인딩(dynamic binding)이다.
    · 상속성: 클래스는 상속이 가능하다.
    · 코드의 재활용: 상속이 가능하다는 것은 여러 곳에서 재활용 할 수 있다는 의미이다.

    객체 지향 5대 원칙(SOLID)

    · 객체 지향 프로그래밍의 근간
    · 느슨한 결합, 높은 응집력이 중요함

    단일 책임 원칙
    Single Responsibility Principle
    • · 클래스는 단일 기능만 구현하도록 할 것
    • · 예를 들어 게임 플레이어에 이동, 소리 기능을 넣는 것이 아니라
    • · 이동 클래스, 소리 기능 클래스를 따로 구현하고 그 클래스들 붙이는 것

    리스코프 치환 원칙
    Liskov's Substitution Principle
    • · 파생 클래스가 기본 클래스를 대체할 수 있어야함
    • · 예를 들어, 기본 클래스가 탈것이고 전후좌우 이동 함수를 구현했다고 하자
    • · 기본 클래스를 상속한 파생 클래스는 자동차, 자전거, 기차가 있다고하자
    • · 자동차, 자전거는 이해가되지만, 기차는 직진후진만 가능하다 그러면 기본 클래스의 좌우 함수는 사용할 수 없다.
    • · 즉 함수를 비워두거나, 오류가 발생할 것이다.
    • · 더 직관적으로 자동차를 상속 받아 여러 자동차는 괜찮지만, 자동차를 상속 받아 비행기를 만드는 것은 틀렸다.
    • · 따라서 회전 가능, 이동 가능 등 인터페이스를 더 세분화하여 만들고, 그 인터페이스를 자동차 탈것, 기차 탈것 클래스가 상속 받는 형식 알맞다.
    인터페이스 분리 원칙
    Interface Segregation Principle
    • · 인터페이스에 모든 기능을 통합하지말고, 최대한 나눠서 인터페이스를 구현하라는 것
    • · 예를들어 플레이어 상태 인터페이스에 이동, 스텟, 데미지 등 모두 넣는 것이 아니라, 이동, 스텟, 데미지 인터페이스로 나누어 여러개 만들고
    • · 특정 플레이어가 필요한 인터페이스만 상속 받는것, 이렇게 하면 코드 결합도가 낮아지고 수정이 용이해짐
    의존성 역전 원칙
    Dependency Inversion Principle
    • · 느슨한 결합
    • · 예를 들어 Door, switch 클래스가 있다고 하자
    • · switch 클래스가 Door 클래스를 내포 클래스로 가진다면
    • · switch 클래스는 Door 클래스만 여닫는 기능을 구현할 수 있다 (강한 결합)
    • · switch와 Door 클래스 사이에 ISwitchable이라는 인터페이스 클래스를 만들어서
    • · switch와 Door 클래스 모두가 ISwitchable이라는 인터페이스 클래스를 상속한다면
    • · switch 클래스는 ISwitchable 인터페이스 객체를 거쳐 Door 클래스의 여닫는 기능을 구현할 수 있다. (느슨한 결합)
    • · 이렇게 하면 switch는 ISwitchable 클래스를 상속받은 모든 객체에 대한 통제를 할 수 있게 된다.

    접근제한자

    이름 설명
    public
    • · 클래스 외부에서 읽기 쓰기 가능
    • · 클래스의 객체와 프로그램 사이의 인터페이스 역할
    • · 설계의 추상화가능 세부적인 구현을 추상화(private)와 분리하는 것을 캡슐화(encapsulation)라고 함
    • · 데이터 은닉은 캡술화의 한 예
    • · 즉, 클래스는 기본적으로 private에는 추상화한 것(사람이라면 눈, 코, 입)
    • · public에는 추상화 한것의 세부 기능(눈 깜빡임, 코로 냄새 맡기, 입으로 말하기)
    private
    • · 클래스의 디폴트 접근 제한자(참고로 구조체는 public이 기본 접근 제한)
    • · 클래스 내부에서만 읽기 쓰기 가능
    • · 외부에서 임의로 접근 불가능하게 함, 데이터 은닉(data hiding)
    • · 데이터의 무결성을 지킬 수 있으므로 좋은 것
    protected
    • · 상속 받은 클래스에서 읽기 쓰기 가능

    멤버

    이름 설명
    멤버 변수

    · 일반적으로 private에 선언
    · 데이터 멤버라고도 함

    멤버 함수

    · 일반적으로 public에 선언
    · 멤버 함수는 정의할 때, 사용 범위 결정 연산자가 있어야 함, 그 함수가 어느 클래스에 속하는지 나타내야한다.
    · 멤버 함수 안에서는 클래스의 private 멤버에 접근 가능
    · 멤버가 아닌 함수안에서 클래스 private 멤버에는 접근 불가, 하지만 friend 함수는 가능
    · 클래스 선언 안에 멤버 함수 정의가 있다면 자동으로 인라인 함수가 됨
    · 코드 수정 규칙(rewrite rule)에 따르면 이는 선언 안의 정의는 원형으로 대체되고, 클래스 선언 바로 뒤에 정의가 추가되는 것과 같음
    · 클래스 선언 밖에 정의한 경우도 멤버 함수 선언 앞에 inline 키워드를 붙이면 인라인 함수가 됨

    생성자와 소멸자

    디폴트 생성자

    · 디폴트 생성자 3가지
    · 아예 정의하지 않으면 자동 생성
    · 매개변수가 없는 생성자를 직접 오버로딩
    · 디폴트 매개변수만 있는 생성자를 직접 오버로딩
    · 디폴트 생성자는 클래스에 단 하나만 존재해야함, 만약 저 방법을 다 써서 만들면 안됨
    사용자 정의 생성자

    · 사용자가 직접 정의한 생성자
    · 생성자는 반환형이 아예 없음(void도 X) 클래스 이름과 동일한 이름으로 작성
    · 일반적으로 멤버 변수는 private에 선언하므로 객체 생성 후 객체를 통해 초기화 불가
    · 객체를 성공적으로 초기화 하기위한 적당한 멤버 함수가 바로 생상저임
    · 생성자 내부에 멤버 변수 초기화, 파라미터를 정의하여 직접 값을 전달, 디폴트 파라미터를 정의하거나 하면 됨
    · 생성자를 사용하는 2가지 방법

  • · 명시적 호출
    • A a = A();
    A* a = new A();

  • · 암시적 호출
    • A a; // A a(); 이거는 틀림! 주의 이거는 함수 선언표현이되어버림
    A* a = new A;
    A* a = new A;
    		cs
    · 객체의 동적 메모리 대입 (코드 추가)

    변환 생성자

    · 사용자 정의 생성자 중 매개 변수가 하나인 생성자
    · 내장 데이터형을 클래스형으로 변환 할 때 사용(반대의 경우는 변환 함수)
    · 암시적 호출의 문제점을 방지하기 위해 explict 키워드를 선언 앞에 작성하자
    · explicit 키워드를 작성하면 암시적 형 변환을 막는다.

    소멸자

    · 소멸자와 virtual 키워드
    · 소멸자는 반드시 가상화되어야한다.
    · 업 캐스팅의 경우 소멸자가 가상화되 있지 않으면 포인터형에 의거해 소멸자가 호출되어 파생 클래스 소멸자가 호출되지 않는다.
    · 따라서 기본 클래스 소멸자를 가상화 해야 파생 클래스의 소멸자가 호출되고 그 이후 자동으로 기본 클래스 소멸자가 호출되어 완전 소멸된다.

    변환 함수

    · 클래스형을 내장 데이터형으로 변환할 때 사용 (반대는 변환 생성자)
    · 암시적으로 호출될 경우 문제가 생길 수 있으므로 explicit 키워드를 작성해야한다.
    · 아니면 차라리 일반함수를 명시적으로 호출하는 형태가 확실히 안전하다.

    연산자 오버로딩 (operator overloading)

    · 클래스 선언에 연산자 오버로딩 함수를 작성하여 클래스 형 객체에 맞는 연산자를 사용자가 직접 정의할 수 있다.
    · 사용자가 직접 정의할 수 있으나, 마음대로하지말고 연산자의 본래 의미에 부합하도록 정의해야한다.
    · 연산자 오버로딩은 오리지널 연산자 문법 규칙을 위배 불가하다. 예를들어 단항 연산자를 이항 연산자로 바꿀 수 없다.
    · 연산자 기호를 새로 만들수 없다.
    · sizeof, .(멤버 연산자), *, ::, ?:, typeid(RTTI연산자), const_cast, dynamic_cast, reinterpret_cast, static_cast는 오버로딩 할 수 없다.
    · =(대입 연산자), ()(함수 호출 연산자), [](배열 인덱스 연산자), ->(클래스 멤버 접근 포인터 연산자)는 멤버 함수로만 오버로딩해야한다.

    오버로딩 가능한 연산자들

    멤버 함수와 virtual 키워드

    · virtual은 가상, 즉 가짜라는 뜻이다. 멤버 함수 선언 앞에 작성한다.
    · virtual 함수의 경우 컴파일러는 이 함수를 가상화하여 정적 바인딩 하지 않고 동적 바인딩 한다.
    · (컴파일러: 어? 얘는 가짜(virtual)네? 나한테는 가짜잖아? 가짜는 건드리지말고 런타임으로 넘겨버려야지!(͠≖ ͜ʖ͠≖)👌 )
    · 따라서 가상 함수의 경우 컴파일 타임이 아니라 런 타임에 호출이 결정된다.
    · 기본 클래스 멤버 함수가 파생 클래스에서 오버라이드 되었고
    · 기본 클래스 포인터 혹은 참조에 파생 클래스 포인터나 참조가 대입될 경우
    · 파생 클래스에서 오버라이드된 기본 클래스 멤버 함수의 선언 앞에 virtual 키워드가 작성되어있다면
    · 대입되는 파생 클래스 객체에 따라서 다른 오버라이된 함수가 호출된다. 따라서 동적 바인딩이다.
    · 이는 SOLID 원칙을 지킬 수 있도록 해주는 중요한 기능 중 하나이다.
    · 이 기능이 있기 때문에 클래스로 인터페이스 구현이 가능하다.

    추상 클래스 (ABC: abstract class)

    · 순수 가상 함수를 하나 이상 가진 클래스는 추상 클래스가 된다.
    · 순수 가상 함수는 함수 원형 뒤에 = 0을 작성한다.
    · 순수 가상 함수는 반드시 파생 클래스에서 재정의 되어야한다.
    · 추상 클래스는 인스턴스화 될 수 없다. 왜냐하면 기본 클래스(부모)의 '역할'만을 위해 존재한다.
